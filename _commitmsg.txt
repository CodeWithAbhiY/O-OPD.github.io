feat(server): production-grade backend foundation + SQLite (Phase 1)

Modular Express architecture (config/db/middleware/schemas/services/
controllers/routes) with strict standards applied:

- Database: real SQLite via node-sqlite3-wasm (no native build needed).
  Schema with PRIMARY KEYs, NOT NULL/CHECK constraints, FOREIGN KEYs,
  indexes, and a PARTIAL UNIQUE INDEX to block active double-booking while
  allowing re-booking of cancelled slots. Idempotent migrate + transactional seed.
- Validation: Zod schemas via reusable validate() middleware; query/params
  validated, unknown body keys rejected (anti mass-assignment).
- Security: parameterized statements (no SQL injection), helmet, CORS allowlist,
  10kb body cap, centralized error handler that never leaks internals.
- API: GET /api/doctors (specialty/location filter, pagination, sort) and
  /api/doctors/:id, returning a typed { data, pagination } contract.
- Config via env (Zod-validated); structured logging; audit_log table ready.

Front-end (results.js): consume the new { data, pagination } contract, map to
the internal shape, and drop the fake "distance" field/sort. Graceful fallback
to bundled sample data remains for GitHub Pages.

Tables for users/bookings/audit are created now so Phase 2 (auth) and Phase 3
(bookings) drop in without re-migrating.
